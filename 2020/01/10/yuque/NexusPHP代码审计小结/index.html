NexusPHP代码审计小结

发表于 2020-01-10

分析

这套 cms，存在全局过滤 sql 注入的函数

用 mysql 防注入函数转义，并且用单引号包裹住语句，导致无法注入引号，也无法引入变量（单引号中的$不会被识别成变量）
并且常常有整型转换

0x01 iconv 的截断

我们看下 php 官方的文档，关于 iconv 这个函数，官方是怎么说的

在 PHP < 5.4.0，字符非法时会被截断，只返回非法字符串之前能正常解码的内容
在 PHP >= 5.4.0 时，字符非法时会报错返回 False，除非输出字符串里拼接了//IGNORE，本套 cms 就是这种情况

也就是说，我们可以控制输出，利用特性 bypass 文件后缀之类的检测

0x02 nowarn.php 存在 sql 注入

需要登录且不为普通用户


nowarn.php 的第 36 行开始，直接拼接进 sql 语句了

0x03 反序列化

全局搜索 unserialize，跟进结果，发现可控点被 sqlesc 给过滤了，md 我单双引号怎么说嘛。。。
最骚的是全局搜索 class 和 public function 根本没有结果，呵呵哒，一个功能类的没有，我真是醉了。
ref：https://shavchen.github.io/PHP%E5%8F%8D%E5%BA%8F%E5%88%97%E5%8C%96%E6%BC%8F%E6%B4%9E/
https://www.anquanke.com/post/id/86452

总结

  1. 这套 cms 审下来，有好很多地方都是形如0x02 的注入一样的拼接，但都需要高权限才能访问到漏洞代码处，进而注入，价值不大
  2. 一个 php 的类读没有，参数绝大部分都来强制类型转换，没有常见的命令执行函数
  3. preg_match 带/e 的模式，可控参数有类型限制，无果
  4. 可以参考这几个老哥的cve和cnnvd，17 年的时候刷了一波大规模的洞，主要是 xss 和 sql 注入，个人觉得索然无味
  5. 这次审计也只发现了 nowarn.php 中没有报送的一个参数，不过需要高权限账号，没有多大利用意义就是了
